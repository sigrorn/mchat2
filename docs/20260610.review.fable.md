# Project Review — 2026-06-10 (Claude Fable 5)

Full-project review covering architectural debt, tech debt, security, and
choice of language/frameworks. Method: four parallel review passes (one per
area), followed by manual verification of every load-bearing claim (lint run,
capability/CSP config, git tracking state, open issues). Findings each carry
a stable kebab-case tag, an **Impact** score (1–5: blast radius if it bites)
and a **Severity** score (1–5: likelihood of biting × cost of fixing late).
The [Risk overview](#risk-overview-ordered-by-impact--severity) at the end
orders everything by Impact × Severity.

Verified ground truth this review is based on:

- `npm run lint` **fails at HEAD** (1 error, `src/lib/tauri/sql.ts:10`).
- `tauri.conf.json:23` sets `"csp": null`; capabilities grant
  `fs:allow-home-read-recursive`, `fs:allow-home-write-recursive`,
  `shell:default`, and runtime-extendable HTTP scope.
- Issue #115 (open since 2026-04-21): "[deferred] Tighten Tauri capabilities
  and set production CSP before release".
- Issue #296 (open): "//pop can fail with `database is locked`" — the
  locking discipline has a live, unresolved failure mode.
- Only `.build-counter.json` is an erroneously tracked artifact; the
  compiled `*.config.js` / `*.tsbuildinfo` files are ignored but present on
  disk.
- No CI exists (no `.github/workflows/`).

## 1. Architectural debt

### arch-lint-boundary-self-trip — boundary lint rule trips on its own shim (lint red at HEAD)

**Impact 3 · Severity 4 · Score 12**

The #142 layer-boundary rule in [eslint.config.js:52-78](../eslint.config.js#L52-L78)
applies `no-restricted-imports` (`@tauri-apps/*`) to **all** of `src/lib/**`
with no carve-out for `src/lib/tauri/**` — the one directory that is
*supposed* to import `@tauri-apps/*` (ARCHITECTURE.md §"What runs where").
When #296 rewrote [src/lib/tauri/sql.ts](../src/lib/tauri/sql.ts#L10) to use
`invoke` from `@tauri-apps/api/core`, the rule fired and `npm run lint`
(`--max-warnings=0`) has been red since. A dead lint gate means every other
boundary rule is now unenforced in practice — violations land invisibly.
Fix is small: an override block exempting `src/lib/tauri/**` from the
`@tauri-apps/*` pattern. Worth checking at the same time why the *other*
shim files (`keychain.ts`, `http.ts`, …) don't trip the rule — if the glob
pattern only matches single-segment specifiers, the rule is also weaker than
intended.

### arch-locking-discipline-by-convention — three-tier locking rests on developer discipline, and still has a live bug

**Impact 4 · Severity 3 · Score 12**

The transaction model (ADR 011; ARCHITECTURE.md §"Transactions and locking
rules") asks every contributor to correctly pick between `transaction()`,
`withSerializedSection`, and plain queued calls, and to remember to thread
`txn.db` / `reposFor(txn.db)` inside transaction bodies. There is **no
type-level enforcement**: a plain `repo.foo()` inside a body compiles fine
and deadlocks at runtime (waiting on the queue head the section holds). The
guard in [transaction.ts](../src/lib/persistence/transaction.ts) only catches
nested/overlapping `transaction()` calls. Evidence that this is not
theoretical: **issue #296 is still open** ("//pop can fail with `database is
locked`") even after the single-connection SQLx bridge landed (commit
84416eb). The design is genuinely clever and well-documented, but it is the
single most fragile load-bearing construct in the codebase: every future
multi-statement write is a chance to get it wrong, and failures manifest as
intermittent `database is locked` errors rather than test failures.

### arch-dual-cache — two cache layers (Zustand stores + repo-query cache) with manual synchronization

**Impact 3 · Severity 3 · Score 9**

ARCHITECTURE.md §"State management" documents four different cache-update
patterns (optimistic, persist-then-update, cache-only patch, reload) across
two overlapping cache layers ([useRepoQuery](../src/lib/data/useRepoQuery.ts)
+ per-store Zustand caches). Mutations must update the right one(s); nothing
enforces it. This is acknowledged transitional state (the data/UI-state
separation is step 2 of the to-mchat3 roadmap), but until the migration
completes, every new mutation is a chance for a stale-cache bug, and the
#279-style "narrow cache-only setter" fixes show this class of bug has
already occurred repeatedly.

### arch-dbi-vs-reposfor — two coexisting patterns for the same problem (threading transaction handles)

**Impact 2 · Severity 2 · Score 4**

Repo functions take an optional `dbi: Kysely<Database>` (ADR 011 pattern,
~27 functions) while [repoContext.ts](../src/lib/persistence/repoContext.ts)
provides the successor `reposFor(txn.db)` bundle (~12 call sites). The
coexistence is explicitly documented as a pending cleanup
(repoContext.ts:21-27). Until the optional-dbi form is removed, both
patterns must be understood, and the optional parameter remains forgettable
(silently routes through the global queue → deadlock; see
arch-locking-discipline-by-convention).

### arch-visibility-dual-write — visibility matrix dual-written to JSON column and relational table

**Impact 2 · Severity 2 · Score 4**

`persona_visibility` (relational, slug-keyed) is the read path;
`conversations.visibility_matrix` (JSON, id-keyed) is dual-written "for
rollback safety" (#202; ARCHITECTURE.md). Additionally,
[handlers/visibility.ts:46-47](../src/lib/commands/handlers/visibility.ts#L46-L47)
has `handleVisibilityDefault` write the matrix twice (the rebuild persists
both layers, then `setVisibilityMatrix` dual-writes the JSON column again) —
the #279 cache-only-setter pattern (`applyVisibilityMatrixCache`) is not yet
exposed through `CommandDeps`. Harmless today, but dual-writes that outlive
their rollback window become permanent schema confusion; needs an explicit
drop-the-JSON-column migration.

### arch-god-components — several components past the project's own ~400-line threshold

**Impact 3 · Severity 2 · Score 6**

[PersonaPanel.tsx](../src/components/PersonaPanel.tsx) is 950 lines (was 855
when docs/IfIWereTostartOver.md already flagged it),
SettingsOpenaiCompatTab.tsx 564, FlowEditor.tsx 482, MessageList.tsx 474,
PersonaFormFields.tsx 407. The retrospective's own diagnosis (Tailwind
class-soup readability wall at ~400 LOC) still applies, and PersonaPanel has
grown ~11% since it was written down. Acknowledged debt; the risk is review
quality and regression rate in the most-edited UI surfaces.

### arch-app-tsx-import-mix — mixed static/dynamic imports in App.tsx (tracked: #300)

**Impact 1 · Severity 1 · Score 1**

`httpScope` and benchmark warmup are both statically and dynamically
imported, producing Vite build warnings. Already tracked as open issue #300
(good-first-issue, fold into next change). Listed for completeness.

### Positive validations (no action)

- **Layering holds.** Components→stores→use-cases→repos→tauri-shim layering
  is real, ESLint-enforced (modulo arch-lint-boundary-self-trip), and no
  back-imports were found.
- **Deps pattern works as designed** (ADR 005): 17 atomic + 7 composed
  `*Deps` shapes in [deps.ts](../src/lib/app/deps.ts) — verbose but
  deliberate, and it keeps use cases testable without React/Zustand.
- **No Zod/Kysely schema duplication**: Zod stays at trust boundaries
  (ADR 003), `schemaParity.test.ts` pins migrations↔schema agreement.
- **Documentation is exceptional**: ARCHITECTURE.md, 14 ADRs, retrospective,
  and a rewrite-vs-refactor analysis that reaches defensible conclusions.

## 2. Tech debt

### debt-no-ci — no CI pipeline; all quality gates are manual (and one is currently red)

**Impact 4 · Severity 3 · Score 12**

There is no `.github/workflows/` (and no other CI config). Lint,
type-check, the 203-file test suite, and the build are run only when someone
remembers. The proof this matters is already in the repo: lint has been
failing at HEAD since #296 landed (see arch-lint-boundary-self-trip) and
nothing noticed. For a solo project the cost of a minimal workflow (lint +
`tsc -b` + `vitest run` on push) is one file; it converts the project's
excellent test culture from opt-in to guaranteed.

### debt-config-shadowing — compiled `vite.config.js` can shadow `vite.config.ts`

**Impact 2 · Severity 3 · Score 6**

`tsc -b` emits `vite.config.js`, `vitest.config.js`, `playwright.config.js`
(transpiled, with `__awaiter`/`__generator` helpers) next to their `.ts`
sources. They are git-ignored (verified: not tracked), but they sit on disk
— and Vite's config resolution tries `vite.config.js` **before**
`vite.config.ts`. `npm run build` is safe (`tsc -b` regenerates the .js
first), but `npm run dev` after editing `vite.config.ts` without an
intervening `tsc -b` silently runs the **stale** compiled config. Fix:
exclude the config files from the emitting tsconfig (or set `noEmit` for
them), and delete the strays.

### debt-build-counter-tracked — `.build-counter.json` is committed and mutated by every version bump

**Impact 2 · Severity 3 · Score 6**

`git ls-files` confirms [.build-counter.json](../.build-counter.json) is
tracked; `scripts/bump-version.mjs` rewrites it on every bump. On any
branching workflow this guarantees merge conflicts in a metadata file, and
on CI it dirties the tree. Either commit it intentionally as versioned state
(current behavior, then this is a wont-fix) or `git rm --cached` it and let
the bump script regenerate. Related, informational: the issue-number-based
scheme (`MAJOR=floor(issue/100)`) means the next issue (#301+) flips the
version to 3.x — intentional per `bumpLogic.mjs`, but worth being conscious
of before it happens.

### debt-silent-catches — eight `.catch(() => {})` / empty-catch sites, some unexplained

**Impact 3 · Severity 2 · Score 6**

Justified, commented cases exist (App.tsx warmups #299/#166,
`modelWarm.ts` best-effort refresh, `http.ts:176` reader-cancel cleanup).
Unexplained ones: [models.ts:268](../src/lib/providers/models.ts#L268)
(model-cache update can fail silently → stale model lists with no signal)
and [SettingsOpenaiCompatTab.tsx:202](../src/components/SettingsOpenaiCompatTab.tsx#L202)
(HTTP-scope registration failure is swallowed → a custom provider then
fails later with an opaque scope error). The project already built
`backgroundTask(label, fn)` specifically because silent fire-and-forget
failures were a recurring debugging headache (#270/#279) — these sites
should use it or carry a why-silent comment.

### debt-dependency-lag — one-major-behind across several dependencies

**Impact 2 · Severity 2 · Score 4**

zod 3 (v4 out), tailwindcss 3 (v4), vite 5 (v6/7), vitest 2 (v3),
react-markdown 9 (v10), eslint flat-config on 9 (fine). Nothing here is
urgent or vulnerable; the risk is the usual compounding cost of multi-major
jumps if deferred for years. `isomorphic-dompurify` could be plain
`dompurify` (browser-only webview) — minor. Rust side is healthy (tauri 2,
sqlx 0.8, keyring 3, all maintained).

### Positive validations (no action)

- **Repo hygiene is otherwise good**: dist/, target/, test artifacts all
  ignored and untracked; no secrets found anywhere (src, tests, docs,
  traces/ is empty); traces and test-results not committed.
- **Lint/type discipline**: `strict` + `noUncheckedIndexedAccess` +
  `exactOptionalPropertyTypes`; `no-explicit-any` as error; only ~6
  eslint-disable directives, all justified.
- **sql.js correctly dev-only**; mermaid/viz correctly lazy-loaded into
  separate chunks ([vite.config.ts](../vite.config.ts) manualChunks).
- **bump script is tested** and never-goes-backwards; the hand-written
  `bumpLogic.d.mts` is a reasonable trick to type-check `.mjs` imports.

## 3. Security

Threat model: single-user desktop app; the realistic adversary is a
**malicious/compromised LLM response** rendered in the webview, plus local
data-at-rest exposure. In a Tauri app, renderer XSS is qualitatively worse
than browser XSS: the webview can `invoke()` every registered command.

### sec-webview-hardening-gap — CSP disabled + broad capabilities turn any renderer XSS into key theft and file access

**Impact 5 · Severity 3 · Score 15** — *tracked but deferred as #115 since April*

Confirmed configuration:

- [tauri.conf.json:23](../src-tauri/tauri.conf.json#L23): `"csp": null`.
- [capabilities/default.json](../src-tauri/capabilities/default.json):
  `fs:allow-home-read-recursive` + `fs:allow-home-write-recursive` (lines
  49-50) — webview can read/write the entire home directory, the
  additional `fs:scope` of `$APPDATA/**` notwithstanding; `shell:default`
  (line 42) — webview can `shell.open()` arbitrary URLs/paths;
  keychain commands (`keychain_get/set/remove/list`,
  [keychain.rs:85-87](../src-tauri/src/keychain.rs#L85-L87)) and
  `register_http_hosts` are invokable with no narrowing permission.

The chain: one XSS in the rendering pipeline → `invoke("keychain_list")` →
`keychain_get` per provider → `register_http_hosts(["https://attacker.tld"])`
([http_scope.rs:49-75](../src-tauri/src/http_scope.rs#L49-L75) accepts any
string; only the *frontend* validates with `new URL()`) → `fetch` exfiltrates
every API key; or read/write any file under `$HOME`. The XSS prerequisite is
real but narrow today — react-markdown is safe-by-default, no `rehype-raw`,
and the single `dangerouslySetInnerHTML`
([DiagramBlock.tsx:41](../src/components/DiagramBlock.tsx#L41)) is fed
through DOMPurify (`sanitizeSvg`) — so the *current* exploitability hinges
on a DOMPurify bypass or a future rendering change. That is exactly what
defense-in-depth is for, and every layer of it is currently switched off.
Recommended (matches #115): set a restrictive CSP; drop
`fs:allow-home-*-recursive` in favor of the `$APPDATA` scope plus dialog-
mediated access; replace `shell:default` with an https/http-only open
permission; validate scheme/host in `register_http_hosts` on the Rust side;
consider gating `keychain_list`.

### sec-gemini-key-in-url — Gemini API key sent as URL query parameter

**Impact 3 · Severity 3 · Score 9**

[gemini.ts:29](../src/lib/providers/gemini.ts#L29) builds
`...:streamGenerateContent?alt=sse&key=<apiKey>`. Keys in URLs leak into
anything that logs URLs (proxies, corporate TLS middleboxes, OS-level
diagnostics, future trace tooling — the redaction patterns in
[redact.ts](../src/lib/security/redact.ts) target `Bearer`-style headers,
not query strings). Google supports the `x-goog-api-key` **header** for the
same API; switching is a small, compatible change and removes the whole
class.

### sec-mermaid-securitylevel-implicit — mermaid initialized without explicit securityLevel

**Impact 2 · Severity 2 · Score 4**

[diagramRenderer.ts:40](../src/lib/rendering/diagramRenderer.ts#L40) calls
`mermaid.initialize({ startOnLoad: false, theme: "neutral" })` and relies on
(a) mermaid's default `securityLevel` and (b) the post-render `sanitizeSvg`
DOMPurify pass. The two-layer defense is sound today; pinning
`securityLevel: "strict"` makes it survive mermaid upgrades and future
refactors that might drop the sanitize step.

### sec-error-bodies-unredacted — full HTTP error bodies propagate without truncation/redaction

**Impact 2 · Severity 2 · Score 4**

[http.ts](../src/lib/tauri/http.ts#L59-L72) throws `HttpError` with the full
response body; a misconfigured/malicious OpenAI-compat endpoint's error text
flows into notices/traces before any redaction. Truncate (e.g. 1 KB) and
run through `redact()` at the boundary.

### sec-traces-plaintext — trace files and exports are plaintext conversation contents

**Impact 2 · Severity 2 · Score 4**

Trace files (`${workingDir}/debug/`) contain full prompts/responses;
redaction is pattern-based, not guaranteed. Acceptable for a single-user
tool, but worth a README note and a default location under `$APPDATA`
rather than an arbitrary user-chosen working directory.

### sec-compat-baseurl-validation — OpenAI-compat preset baseUrl validated only as non-empty string

**Impact 2 · Severity 2 · Score 4**

[openaiCompatConfig.ts:49](../src/lib/schemas/openaiCompatConfig.ts#L49):
`baseUrl: z.string().min(1)`. Today `originOf()`'s `new URL()` throws on
garbage, but the schema is the documented trust boundary (ADR 003) and
should enforce `z.string().url()` + scheme allowlist itself.

### Verified non-findings

- **No SQL injection surface**: all queries compile through Kysely to
  parameterized statements; the Rust bridge ([sql_bridge.rs](../src-tauri/src/sql_bridge.rs))
  binds parameters via sqlx and constrains the DB path to the app config dir
  with a `..` check.
- **No secrets in repo** (src, tests, docs; `traces/` empty). Keys live in
  the OS keychain, are read at call time, and never enter reactive state.
- **react-markdown path is safe**: no `rehype-raw`, no other
  `dangerouslySetInnerHTML`/`innerHTML` sites beyond the sanitized diagram
  block.
- **No auto-updater** = no update-channel attack surface (and no update
  integrity to design yet).

## 4. Choice of language and frameworks

Overall verdict: **the stack fits the problem**, and the project's own ADRs
(esp. 006/007/008) plus `IfIWereTostartOver.md` have already litigated the
big choices honestly. Findings below are residual risks, not regrets.

### stack-ipc-transaction-model — transactional integrity is reconstructed in JS across an IPC boundary

**Impact 4 · Severity 3 · Score 12** *(same root as arch-locking-discipline-by-convention; counted once in the risk table)*

The "fat webview, thin Rust shell" choice (ADR 008) pushes SQLite
transaction discipline into TypeScript: a global JS promise-queue +
section tokens + a deliberately max-1-connection SQLx pool
([sql_bridge.rs](../src-tauri/src/sql_bridge.rs)) simulate what a single
in-process connection gives you for free. It works and is well-tested, but
it is the part of the stack where the platform is being fought hardest:
`BEGIN`/body/`COMMIT` straddle multiple invokes, a webview crash mid-section
leaves a dangling `BEGIN` (next launch recovers only because SQLite rolls
back on reconnect), and issue #296 shows residual `database is locked`
edges. The contained alternative — moving each *whole transaction* into a
single Rust command (one invoke = one atomic unit) — would eliminate the
queue/section machinery for the few true multi-write transactions while
keeping the testable-TS-logic benefits everywhere else. Worth an ADR-level
revisit if #296-class bugs recur.

### stack-webview-variance — Tauri's OS-webview bet vs. Electron's pinned Chromium

**Impact 3 · Severity 2 · Score 6**

Rendering runs on whatever WebView2/WebKit the user's OS ships. For the
current single-user/self-distribution model this is near-zero risk (and the
payoff — small binary, native keychain, OS-managed updates — is real). It
becomes a compatibility-testing liability only if distribution widens.
ES2022 target is appropriately conservative.

### stack-handrolled-query-cache — custom useRepoQuery instead of TanStack Query

**Impact 2 · Severity 2 · Score 4**

~170 LOC replacing a 13 kB dependency, with dedup, prefix invalidation and a
sync `.get()` that TanStack doesn't offer cleanly (ADR 006). Reasonable
call; the residual cost is manual invalidation discipline (no staleness,
no TTL, no retry) — which feeds the arch-dual-cache risk above. If
invalidation bugs recur after the data/UI-state separation completes,
adopting TanStack Query is a contained swap (hook signature compatible).

### stack-sqljs-test-divergence — tests run a different SQLite build than production

**Impact 2 · Severity 2 · Score 4**

sql.js (emscripten, in-memory, no WAL) in tests vs. SQLx/SQLite (WAL,
busy_timeout=5000) in production. Schema drift is eliminated
(`schemaParity.test.ts`, real migrations in tests — genuinely high-ROI
design), but **lock-contention semantics are untestable in the unit suite**
— precisely the class where the open bug (#296) lives. Any future locking
work needs e2e or Rust-side tests, not Vitest.

### Positive validations (no action)

- **Tauri 2 over Electron**: right call for a single-user local-first tool
  (bundle size, native keychain via `keyring`, single-instance plugin).
- **TypeScript strict-max + Zod at boundaries**: exemplary.
- **Kysely over raw SQL / full ORM** (ADR 007): proportionate; parity test
  kills the hand-written-schema drift risk.
- **SSE handling**: custom ~30-line parser over `ReadableStream` through
  tauri-plugin-http works, handles CRLF normalization and idle timeout; a
  library would not be simpler.
- **React 19 + Zustand, Solid swap deferred** (ADR 008): the deferral logic
  (fix data/UI-state separation first, then re-benchmark) is sound and has
  explicit revisit triggers.
- **OpenAI-compat meta-provider** (ADR 004): native adapters only where
  protocols genuinely diverge; presets + static/runtime HTTP scope (ADR 012)
  is the right shape (validation gap noted under security).

### Suggested order of attack

1. **Fix the lint exemption** (arch-lint-boundary-self-trip) — minutes, and
   it un-breaks every other gate.
2. **Add minimal CI** (debt-no-ci) — lint + tsc + vitest on push.
3. **Do #115** (sec-webview-hardening-gap) — CSP, narrow fs scope, scheme
   allowlist on shell-open and `register_http_hosts`. It has been deferred
   for two months and is the only score-15 risk.
4. **Gemini header auth** (sec-gemini-key-in-url) — small, self-contained.
5. **Decide the transaction question** (#296 / stack-ipc-transaction-model):
   either chase the remaining `database is locked` edge within the current
   model, or write the ADR for moving whole transactions into single Rust
   commands.
6. Everything score ≤ 6 as opportunistic cleanup, ideally bundled with the
   issues that already track them (#300, dual-write migration, component
   splits per the to-mchat3 roadmap).
